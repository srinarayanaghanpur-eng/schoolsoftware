ACADEMIC YEAR — LOGIN-SELECTION FEATURE (PHASED ROADMAP)
=========================================================

STATUS: Planning only. NOT yet implemented. This document is the build plan
for a later coding session.

---------------------------------------------------------
CONTEXT / WHY
---------------------------------------------------------
Today the app uses a SINGLE GLOBAL "active year" (the `academic_years`
collection has one doc with isActive: true) switched from a top-bar dropdown
that changes the year for EVERYONE. Pages read `activeYear` from
AcademicYearContext and pass `academicYearId` as an OPTIONAL query param.
Management lives at /admin/academic-years (reached via the Exams & Marks
submenu); write APIs are gated to [admin, principal].

GOAL: move to a PER-LOGIN selected academic year. Back-office staff choose a
year on the login screen; that choice scopes their data for the session;
academic-year MANAGEMENT becomes super_admin-only and is removed from normal
navigation.

---------------------------------------------------------
DECISIONS (LOCKED)
---------------------------------------------------------
1. WHO SELECTS AT LOGIN: Admin, Super Admin, Principal, Accountant.
   Teachers and parents/students do NOT select — they use the default
   (isActive) year. Receptionist optional — currently excluded.
2. MODEL: REPLACE global active-year switching with per-login selection.
   isActive remains only as the DEFAULT preselected option in the login
   dropdown. Remove the global top-bar SWITCHER; top bar DISPLAYS (does not
   switch) the selected year.
3. ENFORCEMENT: Client-guard first. Block pages from querying until a year is
   selected; always send academicYearId + limit=25. Backend keeps the filter
   "applied-when-present" so legacy docs without the field are not hidden.
   Hard backend requirement + backfill is a later follow-up.
4. DELIVERY: Phased. Phase 1 reviewed before Phase 2 starts.

=========================================================
PHASE 1 — Login selection + global state + top-bar display
          + super_admin-only management
=========================================================

1a. PUBLIC ACADEMIC-YEARS LIST ENDPOINT (pre-login)
- New route apps/web/app/api/academic-years/public/route.ts (GET, no auth),
  mirroring the public pattern of login-id/check.
- Returns minimal list { id, name, isActive }[] sorted by startDate desc, from
  academic_years. Reuse serialization from the existing route. Consider a short
  in-process cache like the admin route.
- Note: exposes only year labels publicly (acceptable, low sensitivity).

1b. LOGIN SCREEN DROPDOWN + REQUIRED SELECTION
- In apps/web/app/login/LoginClient.tsx (useTeacherLoginController):
  * Fetch public years on mount; add selectedYearId state defaulting to the
    isActive year.
  * Render an Academic Year <select> in BOTH DesktopLoginExperience and
    MobileLoginExperience (they share the controller).
  * Require a selection before submit (disable Sign In / show validation until
    chosen).
  * The year selection only matters for target roles (admin/super_admin/
    principal/accountant). For teacher/parent/student keep it simple: show it,
    default to active year; non-target roles just won't have it enforced
    downstream.
- On successful login (onSubmit, after signInAndResolveRole): persist the chosen
  year to localStorage (e.g. sriNarayana.selectedAcademicYear = { id, name })
  BEFORE router.replace(destinationForRole(role)).

1c. SELECTED-YEAR GLOBAL STATE (replaces global switching)
- In AcademicYearContext.tsx:
  * Add selectedYearId (init from localStorage, fallback activeYear?.id),
    selectedYear, and setSelectedYear(id) that writes localStorage and triggers
    data refresh.
  * Expose selectedYear as the value all pages use for queries (instead of
    activeYear).
  * Keep years/activeYear for the dropdown default and management display.
- Provider is mounted only for back-office (AppShell), which covers the target
  roles. Teacher/portal untouched in Phase 1 per decision.

1d. TOP BAR: DISPLAY, NOT SWITCH
- In AppShell.tsx: replace AcademicYearSwitcher in the header with a read-only
  badge showing selectedYear?.name (keep HeaderDateLabel showing the year).
  Remove/disable the global activateYear-on-change switcher from normal UI.
- On logout (handleSignOut): clear the localStorage selected-year key alongside
  the existing erp-auth-role cleanup.

1e. REMOVE ACADEMIC YEARS FROM NORMAL NAV & RELOCATE MANAGEMENT TO SUPER_ADMIN
- Remove the { href: "/admin/academic-years", label: "Academic Years", ... }
  entry from the Exams & Marks submenu and from activePrefixes/matchPrefixes in
  AppShell.tsx.
- Relocate the management page to a super_admin-only location, e.g.
  apps/web/app/admin/settings/academic-years/page.tsx (move existing
  apps/web/app/admin/academic-years/page.tsx). /admin/settings is already
  super_admin in routeAccess.ts. Optionally add an explicit routePermissions
  entry for the new path.
- Restrict write APIs to super_admin only: update requireRole/permission checks
  in academic-years/route.ts (POST), [id]/route.ts (PATCH/DELETE),
  [id]/activate/route.ts from [admin, principal] -> [super_admin]. GET stays
  broad so the dropdown/list still loads.
- Remove Add/Edit/Delete/Set-Active buttons for non-super_admin (page already
  computes canWrite/canDelete; tighten to role === "super_admin").

1f. CHANGE-YEAR UX (safe refresh)
- If we still allow an in-app year change for super_admin/admin (optional),
  changing selectedYear should refresh dependent data safely (invalidate the
  sessionStorage academic-years cache + re-run page queries). Simplest:
  setSelectedYear updates localStorage and pages re-query on the changed context
  value.

PHASE 1 ACCEPTANCE
- Back-office user sees a year dropdown on login, must pick one, lands scoped to
  that year, sees it in the top bar, and it clears on logout. Academic Years no
  longer appears in the sidebar/submenu; only super_admin can open the
  (relocated) management page and use add/edit/delete/set-active.

=========================================================
PHASE 2 — Enforce academicYearId + schoolId + pagination(25)
          across queries
=========================================================

2a. CLIENT GUARD ("no year -> no query")
- In each back-office list page (students, payments/finance, exams,
  fee-structures, promotions, receipts, attendance, vouchers/bus-finance), gate
  data fetching on selectedYear?.id. Show a "select a year" empty state instead
  of querying when absent.
- Always pass academicYearId=selectedYear.id and limit=25 (or pageSize=25) and
  the resolved schoolId.

2b. STANDARDIZE PAGINATION TO 25 WHERE MISSING
- Use existing readLimit(value, 25, 100) from firestoreReadLogger.ts.
- Add cursor pagination + limit(pageSize+1) (students/payments pattern) to
  endpoints currently unpaginated or over-fetching: admin/exams,
  admin/fee-structures, admin/promotions (drop hardcoded 500), admin/attendance
  (drop hardcoded 200), admin/bus-finance. Add logFirestoreRead where missing.
- Reduce finance defaults (dues/receivables default 500) toward 25 for list
  views (keep higher caps only for explicit export/report paths).

2c. BACKEND "APPLIED-WHEN-PRESENT" (no hard-require yet)
- Keep .where("academicYearId","==",id) applied when the param is present
  (already the case). Do NOT reject missing params yet (protects legacy docs).
  Add schoolId filter analogously where the field exists on the collection.

2d. schoolId RESOLUTION
- Reuse the claim->env->default resolution from apps/web/lib/payrollAccess.ts
  (school_id/schoolId claim -> process.env.SCHOOL_ID -> default). Thread a single
  getSchoolId() helper into query routes.

PHASE 2 ACCEPTANCE
- Every core list endpoint accepts and applies academicYearId + schoolId,
  returns <=25 rows per page with cursor paging, and pages refuse to load data
  until a year is selected.

=========================================================
FOLLOW-UPS (later, not in Phase 1/2)
=========================================================
